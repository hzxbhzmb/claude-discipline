## 每次任务必做（不可跳过）
1. **读 todo/current.md** → 写入任务计划（含日期头 + 达标标准 + 子任务拆解）
2. **读 methodology/_index.md** → 按场景查阅相关方法论
3. **逐个完成、逐个标记** `[x]`（每个必须有工具调用证据） → 全部完成后**验算**（换路径验证达标） → 才汇报

## 任务管理（硬性阻断规则，不可跳过）
- **禁止在未更新 todo/current.md 前执行任何代码修改或文件创建。** 这是阻断性前置条件，没有例外。
- 收到用户任务后的第一步永远是：读取 todo/current.md → 追加任务计划（含子任务拆解） → 然后才能动手。
- 即使是"快速修一个 bug"这种看似简单的任务，也必须先写入 todo/current.md。
- 尽量使用多 agent 并行执行独立的任务项。
- **每完成一个子任务，立即在 todo/current.md 中标记 `- [x]`。** 不要攒到最后一起标记，不要全部做完再回头补。逐个做、逐个标记，这是防止遗漏的核心机制。
- 确认所有任务项完成后，再向用户反馈执行结果。

### 达标标准（开工前必须明确）
- 写入计划时，必须同时写入**达标标准**：这批任务做到什么程度算"完成"
- 达标标准写在日期头下方、子任务列表上方，格式：`> 达标标准：...`
- 如果用户已经明确说了标准（如"能跑通测试"、"页面不报错"），直接记录，不用再问
- 如果用户没说清楚，**必须先问清楚再动手**，不要自己猜

### 任务格式要求
- 每批任务必须有日期头：`## YYYY-MM-DD — 任务简述`
- 日期头下方紧跟达标标准：`> 达标标准：...`
- 子任务用 `- [ ]` / `- [x]` 格式

### 归档规则
- 当 todo/current.md 超过 80 行时，将已完成章节（所有子任务均 `[x]`）按月归档到 `todo/archive/YYYY-MM.md`
- todo/current.md 只保留：进行中、未完成、未来扩展
- 归档时保持原始格式（含日期头），保留可追溯性

### 完成自检（标记 [x] 前必须执行）
- 标记 `[x]` 前，必须有对应的工具调用证据（Edit/Write/Bash 等），纯文字描述不算完成
- 改了文件 → re-read 确认改动生效
- 跑了命令 → 检查输出是否符合预期
- **没有工具调用 = 没有执行 = 不能标记 [x]**

### 验算论证（汇报前必须执行）
- 全部子任务标记 `[x]` 后，**不能直接汇报完成**，必须先做验算
- 验算 = 用与执行不同的路径来证明达标标准已满足：
  - 改了代码说能跑 → 真的跑一下
  - 说修了 bug → 构造原始触发条件，验证不再触发
  - 说重构没改行为 → 跑测试证明行为不变
  - 说加了功能 → 从用户角度走一遍使用流程
- **核心原则：不能用同一个思路检查两遍，必须换路径**
- 验算通过 → 在 todo 中追加 `> ✅ 验算通过：{验算方法和结果}`
- 验算失败 → 修复后重新验算，不要带着问题汇报完成

### 严禁的反模式
- ❌ 收到任务 → 直接写代码 → 做完再补 todo/current.md（事后补账）
- ❌ 一口气做完所有子任务 → 最后统一标记 `[x]`（批量标记）
- ❌ 任务做了一半就向用户汇报"完成了"（虚假完成）
- ❌ 没有工具调用就标记 `[x]`（凭想象完成）
- ❌ 全部标记完就直接汇报，跳过验算（未经验证的完成）

### 正确的工作流
1. 读 todo/current.md
2. 写入计划 + 达标标准 + 子任务拆解（含日期头）
3. 达标标准不清楚 → 先问用户，清楚 → 继续
4. 执行子任务 A → 有工具调用证据 → 标记 `[x]`
5. 执行子任务 B → 有工具调用证据 → 标记 `[x]`
6. ...逐个完成...
7. 全部标记后，执行验算（换路径验证达标标准）
8. 验算通过 → 记录验算结果 → 向用户汇报

## 研究文档（硬性规则，不可跳过）
- **任何研究、分析、调研工作，必须在完成时将产出写入 `research/` 目录下的 .md 文件。** 不可以只在对话中输出研究结论而不保存文件。
- 拆解 todo 子任务时，如果某个子任务涉及研究/分析/调研，必须在该子任务中明确包含"写入 research/"步骤。
- 所有生成的面向用户阅读的文档必须使用 Markdown (.md) 格式，不要用 .txt。
- 所有生成的文档必须用中文撰写，专有术语可保留英文。

### 严禁的反模式
- ❌ 在对话中输出大段研究结论 → 没有保存到 research/ 文件（研究蒸发）
- ❌ 研究做完后被用户提醒才补写文件（事后补账）

## 错误反思与方法论
- 每次任务中出现的错误（无论是自行发现还是用户反馈），都要在任务结束后进行总结和深度反思。
- 方法论按语义分类存放在 `methodology/` 目录，顶层索引见 `methodology/_index.md`。

### 查阅规则（渐进式披露）
1. 任务开始 → 读 `methodology/_index.md`（轻量索引）
2. 根据当前任务场景 → 只读匹配分类下的详情文件
3. 不要一次性读取所有详情文件

### 写入规则
- 将反思抽象为可迁移的方法论，写入 `methodology/` 对应分类的详情文件
- 写入前先查索引，检查是否可合并到已有条目
- 新分类下首个文件直接创建；若某分类超过 5 个文件，加 `_index.md` 做二级索引

### 定期整理
- 触发时机：某分类新增文件使其超过 5 个，或用户明确要求
- 整理内容：重审分类合理性（拆大合小）、合并重叠条目、抽象具体经验为通用原则、更新 `_index.md` 摘要
