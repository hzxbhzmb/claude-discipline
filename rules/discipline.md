# Claude Discipline 纪律规则

> 本规则由 claude-discipline 插件自动注入，通过 Hook 系统级强制执行。

## 每次任务必做（不可跳过）
1. **读 todo/current.md** → 完成任务三次握手（理解 → 确认 → 授权）
2. **读 methodology/_index.md** → 按场景查阅相关方法论
3. **逐个完成、逐个标记** `[x]`（每个必须有工具调用证据） → 全部完成后**验算** → 四次挥手关闭任务

## 任务三次握手（Hook 系统级强制，不可绕过）

类比 TCP 三次握手：单向发送不够，单向收到也不够，必须确认双方都知道——双方都已经准备好了，而且双方都知道对方准备好了。

**"收到任务就开干"是被禁止的。** Hook 会在你编辑项目文件前检查：当前任务段是否有 `> ✅ 执行授权` 标记。没有 → 系统级阻断。

### 第一次握手：用户发起任务（SYN）

用户给出任务意图。你收到后，在 todo/current.md 新建任务段，记录：

```markdown
## YYYY-MM-DD — 任务简述

**用户意图**：（忠实记录用户说了什么，不加工不缩写）
```

### 第二次握手：AI 回传理解与计划（SYN-ACK）

你不能只回复"收到"或"明白"。你必须**回声式复述**——用你自己的理解重新表达，让用户看到你到底懂了什么、漏了什么、理解偏了什么。

在 todo 中写入：

```markdown
**AI 理解**：
- 目标：我要做什么（用你的话重述，不是复制粘贴用户原话）
- 边界：什么不做 / 什么不能动 / 哪些超出范围
- 路径：我准备怎么做（关键步骤，不需要完整子任务列表）
- 风险：我最担心什么可能出错

> 🤝 待用户确认
```

然后**在对话中向用户确认**你的理解是否正确。

### 第三次握手：用户确认执行授权（ACK）

用户确认后（可能是"对"、"可以"、"方向没问题"、"但 XX 要改一下"等），你在 todo 中记录：

```markdown
> ✅ 执行授权：（记录用户确认的要点，如"用户确认方向正确"或"用户要求增加 XX 约束"）
```

**只有写入 `> ✅ 执行授权` 后，才能开始编辑项目文件。** 这是 Hook 强制检查的。

如果用户要求修正，则更新 **AI 理解** 段，重新走第二次握手。

### 握手完成后，写入达标标准和子任务

```markdown
**达标标准**：...

- [ ] 子任务 1
- [ ] 子任务 2
```

### 为什么需要三次握手

| 风险 | 没有握手时 | 有握手时 |
|------|-----------|---------|
| 解错题 | 用户说 A，AI 理解为 B，做完才发现全错 | 第二次握手暴露理解偏差 |
| 边界越界 | 只让分析，AI 直接改文件 | AI 显式声明边界，用户纠正 |
| 优先级错 | 应该保守修复，AI 走大重构 | AI 声明路径，用户拦截 |
| 路径不认可 | AI 自选方案，用户不买账 | 第三次握手确认路径可接受 |

## 任务管理（硬性阻断规则，不可跳过）
- **禁止在未更新 todo/current.md 前执行任何文件修改或文件创建。** 这是阻断性前置条件，没有例外。
- 收到用户任务后的第一步永远是：读取 todo/current.md → 完成三次握手 → 然后才能动手。
- 即使是"快速修一个 bug"这种看似简单的任务，也必须先握手。
- 尽量使用多 agent 并行执行独立的任务项。
- **每完成一个子任务，立即在 todo/current.md 中标记 `- [x]`。** 不要攒到最后一起标记，不要全部做完再回头补。逐个做、逐个标记，这是防止遗漏的核心机制。

### 任务格式要求
- 每批任务必须有日期头：`## YYYY-MM-DD — 任务简述`
- 日期头下方依次是：**用户意图** → **AI 理解** → `> ✅ 执行授权` → **达标标准** → 子任务列表
- 子任务用 `- [ ]` / `- [x]` 格式

### 归档规则
- 当 todo/current.md 超过 80 行时，将已完成章节（所有子任务均 `[x]`）按月归档到 `todo/archive/YYYY-MM.md`
- todo/current.md 只保留：进行中、未完成、未来扩展
- 归档时保持原始格式（含日期头），保留可追溯性

## 证据链（Hook 系统级强制，不可绕过）

插件通过 Hook 记录**每一次工具调用**（工具名、操作目标、时间戳）到会话证据日志。这是硬数据，你无法伪造。

### 标记 [x] 的证据要求
- 标记 `[x]` 前，**自上次标记以来必须有非 todo 的工具调用**
- Hook 会在你标记 `[x]` 时自动检查证据日志——如果没有找到工具调用痕迹，**操作会被系统拒绝**
- 什么算有效证据：Read 读了相关文件、Edit/Write 修改了文件、Bash 跑了命令、Grep 搜了内容、MCP 工具调用等
- 什么不算证据：只编辑了 todo/current.md 本身、只编辑了 methodology/_index.md

### 验算的证据要求
- 写 `> ✅ 验算通过` 前，**自最后一个 [x] 以来必须有读取/执行类工具调用**
- Hook 会在你写验算行时自动检查——如果没有找到验算证据，**操作会被系统拒绝**
- 验算 = 用工具去拿一手数据确认达标标准已满足
- 不能只凭记忆或推理声称验算通过——必须有工具调用作为证据

### 证据链的意义
- 你的每一步操作都有迹可查——用户可以随时回溯证据日志
- Hook 的检查是系统级的，你绕不过去
- 这不是不信任你，而是让"做了就有痕迹、没做就标不了"成为结构性保证

## 任务四次挥手（关闭协议）

类比 TCP 四次挥手：任务不能"AI 自己说 done 就结束"。

### 第一次挥手：AI 声明完成

全部子任务标记 `[x]` 后，用工具执行验算，确认达标标准已满足。

### 第二次挥手：AI 提交完成证据

在 todo 中写入验算结果：
```markdown
> ✅ 验算通过：{验算方法和结果}
```

同时在对话中向用户汇报：
- 做了什么
- 验算了什么
- 还有什么已知局限或未覆盖的边界

### 第三次挥手：用户验收

用户审查产出。可能：
- 确认完成 → 任务关闭
- 要求补充 → 回到执行阶段
- 发现方向偏了 → 可能需要重新握手

### 第四次挥手：任务正式关闭

用户确认后，任务段视为关闭。如有方法论沉淀，写入 methodology/。

### 验算具体做什么
- 验算 = 用工具去拿一手数据确认达标标准已满足：
  - 改了代码说能跑 → 真的跑一下
  - 说修了 bug → 构造原始触发条件，验证不再触发
  - 说重构没改行为 → 跑测试证明行为不变
  - 说加了功能 → 从用户角度走一遍使用流程
  - 说下载了文件 → ls 确认文件存在且大小合理
  - 说转换了格式 → 读输出文件确认内容完整
- 验算失败 → 修复后重新验算，不要带着问题汇报完成

## 研究文档（硬性规则，不可跳过）
- **任何研究、分析、调研工作，必须在完成时将产出写入 `research/` 目录下的 .md 文件。** 不可以只在对话中输出研究结论而不保存文件。
- 拆解 todo 子任务时，如果某个子任务涉及研究/分析/调研，必须在该子任务中明确包含"写入 research/"步骤。
- 所有生成的面向用户阅读的文档必须使用 Markdown (.md) 格式，不要用 .txt。
- 所有生成的文档必须用中文撰写，专有术语可保留英文。

### 严禁的反模式
- ❌ 在对话中输出大段研究结论 → 没有保存到 research/ 文件（研究蒸发）
- ❌ 研究做完后被用户提醒才补写文件（事后补账）

## 错误反思与方法论
- 每次任务中出现的错误（无论是自行发现还是用户反馈），都要在任务结束后进行总结和深度反思。
- 方法论按语义分类存放在 `methodology/` 目录，顶层索引见 `methodology/_index.md`。

### 查阅规则（渐进式披露）
1. 任务开始 → 读 `methodology/_index.md`（轻量索引）
2. 根据当前任务场景 → 只读匹配分类下的详情文件
3. 不要一次性读取所有详情文件

### 写入规则
- 将反思抽象为可迁移的方法论，写入 `methodology/` 对应分类的详情文件
- 写入前先查索引，检查是否可合并到已有条目
- 新分类下首个文件直接创建；若某分类超过 5 个文件，加 `_index.md` 做二级索引

### 定期整理
- 触发时机：某分类新增文件使其超过 5 个，或用户明确要求
- 整理内容：重审分类合理性（拆大合小）、合并重叠条目、抽象具体经验为通用原则、更新 `_index.md` 摘要

## 严禁的反模式
- ❌ 收到任务 → 直接开干 → 跳过三次握手（会被 hook 阻断）
- ❌ 回复"收到"或"明白"就开始做 → 没有回声式复述理解（不是真正的第二次握手）
- ❌ 收到任务 → 直接写代码 → 做完再补 todo/current.md（事后补账）
- ❌ 一口气做完所有子任务 → 最后统一标记 `[x]`（批量标记）
- ❌ 任务做了一半就向用户汇报"完成了"（虚假完成）
- ❌ 没有工具调用就标记 `[x]`（凭想象完成——会被证据链 hook 阻断）
- ❌ 全部标记完就直接汇报，跳过验算（未经验证的完成——会被 hook 阻断）
- ❌ 验算时不用工具，只凭记忆声称通过（会被证据链 hook 阻断）
- ❌ AI 自己说 done 就结束，不等用户验收（跳过四次挥手）

## 正确的工作流

```
┌─ 三次握手（建立任务连接）──────────────────┐
│ 1. 用户发任务（SYN）                        │
│ 2. AI 回传理解与计划（SYN-ACK）             │
│ 3. 用户确认执行授权（ACK）                  │
└─────────────────────────────────────────────┘
                    ↓
┌─ 执行阶段 ──────────────────────────────────┐
│ 4. 读 methodology/_index.md                 │
│ 5. 执行子任务 → 有工具调用 → 标记 [x]       │
│ 6. ...逐个完成...                           │
└─────────────────────────────────────────────┘
                    ↓
┌─ 四次挥手（关闭任务连接）──────────────────┐
│ 7. AI 用工具执行验算                        │
│ 8. AI 提交验算结果 + 向用户汇报             │
│ 9. 用户验收（确认 / 要求补充 / 重新握手）   │
│ 10. 任务正式关闭 + 方法论沉淀               │
└─────────────────────────────────────────────┘
```
