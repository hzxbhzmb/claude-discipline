# Claude Discipline 纪律规则

> 由 claude-discipline 插件自动注入。Hook 系统级强制执行。

---

## 🚨 BLOCKING REQUIREMENT — 收到任务的第一动作

**用户给你任何任务，你的第一个工具调用必须是用 `Edit` 编辑 `todo/current.md`** 在末尾追加任务段（带本会话 `<!-- session: xxxxxxxx -->` 标注 + `**用户意图**` + `**AI 理解**`），然后**在对话中向用户回声理解、等用户确认后写 `> ✅ 执行授权`**。完成握手前：

- ❌ **不要**用 Read/Grep/Glob 探索代码"先做研究" —— 那是事后补账
- ❌ **不要**在对话里先回答"我打算这么做" —— 那是绕开握手
- ❌ **不要**直接动 Edit/Write/Bash 写项目文件 —— hook 会硬拦
- ❌ **不要**把研究/调研/分析的产出只在对话输出 —— 必须落到 `research/` 目录

**唯一例外**：极轻量任务（单条命令、单文件 ≤10 行小改、纯只读探索）可走快车道——任务段仍要建，但握手压缩为单步 `> ✅ 执行授权：[快车道] {说明}`（详见下文"快车道"）。

如果你不确定该怎么开始，**就用 Edit 编辑 `todo/current.md`**——这是白名单文件，无需任何前置授权即可编辑。

## 严禁的反模式

- ❌ **跳过握手直接开干 / 回"收到"就动手 / 做完再补 todo（事后补账）**
- ❌ **批量标 `[x]`（攒到最后一起标）/ 自己说 done 就结束不等用户验收**
- ❌ **在对话里做研究而不写 `research/` 文件**（"研究/分析/调研"产出必须落地）
- ❌ 验算用和执行相同路径 / 做完才想验算方案
- ❌ 把应走重档的任务（多文件、研究、重构、配置变更）硬塞快车道
- ❌ 用 Bash 写操作绕开握手 / 验算不过 silent 改到过为止

## 工作流

**三次握手（对齐意图）→ 执行（证据链）→ 四次挥手（验算验收）**

1. 读 `todo/current.md` 完成三次握手（理解 → 用户确认 → 授权）
2. 读 `methodology/_index.md` 查阅方法论
3. 逐子任务 → **必须有工具调用** → 标 `[x]`
4. 全部完成 → **反向验算** → 写 `> ✅ 验算通过` → 汇报 → 等用户验收

## 三次握手（强制）

**收到任务就开干 = 禁止。** Hook 检查本会话最新任务段是否含 `> ✅ 执行授权`。

| 步骤 | 做什么 |
|------|-------|
| SYN — 用户发任务 | 在 todo 末尾新建任务段，记 `**用户意图**` |
| SYN-ACK — AI 回理解 | 写 `**AI 理解**`（目标/边界/路径/风险/验算方案）→ 在对话中向用户确认 |
| ACK — 用户确认 | 用户确认后写 `> ✅ 执行授权：{要点}` |

### 任务段标准格式

```markdown
## YYYY-MM-DD — 任务简述 <!-- session: xxxxxxxx -->

**用户意图**：（忠实记录）

**AI 理解**：目标 / 边界 / 路径 / 风险 / **验算方案**（反向路径，与执行不同）

> ✅ 执行授权：{用户确认要点}

**达标标准**：...

- [ ] 子任务 1
- [ ] 子任务 2
```

**新任务段必须追加到 `todo/current.md` 末尾**（最后一个 `---` 之后），不得插中间或顶部。

## 证据链（强制）

每次工具调用自动记录到会话日志。AI 无法伪造。

| 操作 | Hook 检查 | 不通过 |
|------|----------|-------|
| 标 `[x]` | 自上次标记以来有非 todo 的工具调用？ | **拒绝** |
| 写 `✅ 验算通过` | 自最后 `[x]` 以来有读取/执行类工具调用？ | **拒绝** |

有效证据：Read / Edit / Write / Bash / Grep / Glob / MCP。
不算证据：仅编辑 `todo/current.md` 或 `methodology/_index.md`。

## 验算 = 反向验证（核心原则）

**验算不是用同一条路径再检查一遍**——是用不同路径反向验证。类比：3+5=8 的验算是 8-3=5，不是再算一遍 3+5。**验算方案在握手时就规划好**，经用户确认后才执行。

| 执行 | ❌ 同路径重复 | ✅ 反向验证 |
|------|-----------|---------|
| 写代码 | 重读代码 | 跑测试 / 走使用流程 |
| 修 bug | 读改过的代码 | 构造原始触发条件验证 |

## 四次挥手

| 步骤 | 做什么 |
|------|-------|
| AI 验算 | 全部 `[x]` 后按规划方案反向验证 |
| ├ 过 | 写 `> ✅ 验算通过：{方法和结果}` |
| └ 不过 | 写 `> ❌ 验算第 N 次失败：{原因} → 改进：{做了什么}`，重试。**累计超 3 次** hook 强制写 `> ❌ 最终验算失败` 交棒（详见 `troubleshooting.md`） |
| AI 汇报 | 在对话中告知用户 |
| 用户验收 | 确认关闭 / 要求补充 / 重新握手 |

## 快车道（轻量任务豁免）

简单任务（单条命令、单 skill、只读探索、单文件 ≤10 行改动）可压缩握手为单步。

**触发**：用户措辞含 `快速 / 直接 / 小改 / 不用握手 / 一句话 / 跑一下 / 简单`，或任务符合上述特征。

```markdown
## YYYY-MM-DD — 任务简述 <!-- session: xxxxxxxx -->

**用户意图**：{一句话复述}

> ✅ 执行授权：[快车道] {做什么、为什么简单}
```

跳过 AI 理解 / 验算方案 / 子任务列表。完成可加 `> ✅ 完成：{一句话}` 让 auto-archive 自动归档。

**必须走重档**（即使用户说"快速"）：多文件改、研究/调研（产出必须写入 `research/`）、重构、配置/基础设施变更、删除/不可回滚操作、用户质疑你的理解。

## 工具层天然豁免

不触发握手：**Read / Grep / Glob / WebFetch / WebSearch / Skill / MCP**；**Bash 只读**（`ls / cat / grep / find / git status / log / diff / commit / push / fetch / pull` 等）；**项目目录外的 Edit/Write**。

⚠️ **Bash 写操作受握手保护**：`mv / cp / rm / sed -i / 重定向 / git reset --hard / git clean -fd / git checkout -- / git restore / git rm / git mv` 与 Edit/Write 同等对待。

## 多会话并发

同项目多会话共用 `todo/current.md` 时，**任务段标题必须带本会话 session 标注**（`<!-- session: xxxxxxxx -->`）。SessionStart 时插件会告诉你本会话的短 ID。Hook 只认带你 session 标注的段，不连坐祖传段或他会话段。**完整规则**：`${CLAUDE_PLUGIN_ROOT}/rules/concurrency.md`（按需读，hook deny 消息会指）。

## 研究文档（强制）

研究/分析/调研产出**必须写入 `research/` 目录**，不可只在对话输出。子任务文本含 `research/` 时，hook 在标 `[x]` 时检查证据日志中是否真有对 `research/` 的写入。

## 方法论

`methodology/` 目录，顶层索引 `_index.md`。任务开始读 `_index.md`，按场景查阅。反思抽象写入对应分类。某分类超 5 个文件时加二级索引。
