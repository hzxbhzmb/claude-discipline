# Claude Discipline 纪律规则

> 本规则由 claude-discipline 插件自动注入，通过 Hook 系统级强制执行。

## 工作流总览

```
三次握手 → 执行 → 四次挥手
  ↓         ↓        ↓
对齐意图   证据链    验算验收
```

1. **读 todo/current.md** → 完成三次握手（理解 → 确认 → 授权）
2. **读 methodology/_index.md** → 按场景查阅方法论
3. 逐个执行子任务 → **有工具调用** → 标记 `[x]`
4. 全部完成 → **用工具验算** → 写 `> ✅ 验算通过` → 向用户汇报 → 等用户验收

## 三次握手（Hook 强制）

**"收到任务就开干"被禁止。** Hook 检查当前任务段是否有 `> ✅ 执行授权`，没有 → 系统级阻断。

| 握手 | 做什么 |
|------|--------|
| **SYN** — 用户发任务 | 在 todo 新建任务段，记录 `**用户意图**` |
| **SYN-ACK** — AI 回传理解 | 写 `**AI 理解**`（目标/边界/路径/风险/验算方案），写 `> 🤝 待用户确认`，在对话中向用户确认 |
| **ACK** — 用户确认 | 用户确认后写 `> ✅ 执行授权：{确认要点}`。用户要求修正 → 更新理解，重走 SYN-ACK |

握手完成后写 `**达标标准**` 和子任务列表，然后才能编辑项目文件。

### 任务段完整格式

```markdown
## YYYY-MM-DD — 任务简述 <!-- session: xxxxxxxx -->

**用户意图**：（忠实记录，不加工）

**AI 理解**：
- 目标：（用你的话重述）
- 边界：（什么不做）
- 路径：（关键步骤）
- 风险：（最担心什么）
- 验算方案：（怎么用反向路径验证结果——必须与执行路径不同）

> 🤝 待用户确认
> ✅ 执行授权：用户确认方向正确

**达标标准**：...

- [ ] 子任务 1
- [ ] 子任务 2
```

## 证据链（Hook 强制）

每次工具调用自动记录到会话日志（工具名、目标、时间戳）。AI 无法伪造。

| 操作 | Hook 检查 | 不通过 |
|------|----------|--------|
| 标记 `[x]` | 自上次标记以来有非 todo 的工具调用？ | **拒绝** |
| 写 `✅ 验算通过` | 自最后一个 `[x]` 以来有读取/执行类工具调用？ | **拒绝** |

有效证据：Read / Edit / Write / Bash / Grep / Glob / MCP 工具调用。
不算证据：仅编辑 todo/current.md 或 methodology/_index.md。

## 验算 = 反向验证（核心原则）

验算不是"用同一条路径再检查一遍"，而是**用不同的路径反向验证结果**。类比：3+5=8 的验算是 8-3=5，不是再算一遍 3+5。验算方案**必须在三次握手时就规划好**（AI 理解的"验算方案"字段），经用户确认后才执行。

### 正例与反例

| 执行动作 | ❌ 同路径重复（不算验算） | ✅ 反向验证（正确验算） |
|---------|----------------------|---------------------|
| 写了代码 | 重新读一遍代码看"对不对" | 跑测试 / 从用户角度走使用流程 |
| 创建了文件 | `cat` 看内容 | `wc -l` 检查行数 + `grep` 搜关键内容 |
| 修了 bug | 读改过的代码确认改了 | 构造原始触发条件，验证不再触发 |
| 做了研究报告 | 重读自己写的报告 | 拿报告结论回去源文件交叉验证 |
| 下载了 10 个文件 | `ls` 看文件名 | `ls -la` 检查大小 + 抽样 `head` 检查内容非空/非损坏 |
| 重构了代码 | diff 看改了什么 | 跑原有测试验证行为不变 |
| 配置了服务 | 读配置文件 | 实际调用服务验证响应正确 |

## 四次挥手（关闭协议）

| 步骤 | 做什么 |
|------|--------|
| AI 验算 | 全部 `[x]` 后，**按握手时规划好的验算方案**用工具反向验证，写 `> ✅ 验算通过：{方法和结果}` |
| AI 汇报 | 在对话中告知用户：做了什么、验算了什么、已知局限 |
| 用户验收 | 确认关闭 / 要求补充 / 发现方向偏了则重新握手 |
| 正式关闭 | 如有方法论沉淀，写入 methodology/ |

## 任务管理补充规则

- 尽量使用多 agent 并行执行独立的任务项
- **逐个完成、逐个标记 `[x]`**，不要攒到最后批量标记
- 归档：todo/current.md 超 80 行时，将已完成章节按月归档到 `todo/archive/YYYY-MM.md`

## 多会话并发纪律（Hook 强制）

同一项目可能有多个 Claude Code 会话同时工作，共用 `todo/current.md`、`research/`、`methodology/`。为避免互相踩踏：

- **任务段标题必须带本会话 sessionId 短 ID 标注**：`## YYYY-MM-DD — 任务简述 <!-- session: xxxxxxxx -->`。SessionStart 时插件会在规则尾部注入你本会话的短 ID——照抄即可。
- **Hook 只认带你自己 session 标注的任务段**：握手检查、验算检查都按本会话过滤；别的会话和祖传无标注段对你"透明"（不拦截、不连坐）。
- **没带标注 = 没建任务段**：`check-handshake.js` 会拒绝你对项目文件的编辑，并提示把段标题补上 `<!-- session: xxxxxxxx -->`。
- **禁止用 Write 整覆盖 `todo/current.md`**：会吞掉并发会话正在写入的段。必须用 Edit 增量改动——Edit 的 old_string 精确匹配是天然的乐观锁。
- **证据日志按 sessionId 隔离**：`/tmp/claude-evidence-${sessionId}.jsonl` 各自独立；SessionStart 只清自己的或 >24h 陈旧文件，不碰其它活跃会话。
- **升级零摩擦**：SessionStart 会自动把最近一个进行中的祖传段（非归档、无 session 标注、含 `[ ]`）认领给本会话（段标题末尾贴 `<!-- session: xxxxxxxx -->`）。注入的规则里会有"自动认领通知"告诉你认领了哪个——如果不是你要继续的任务，自己用 Edit 把标注挪到别处或删掉。

## 研究文档（Hook 强制）

- 研究/分析/调研的产出**必须写入 `research/` 目录**，不可只在对话中输出
- 拆子任务时，涉及研究的子任务**必须显式包含 `research/` 路径**（如"分析 XX → 写入 research/"）
- **Hook 检查**：标记 `[x]` 时，若子任务文本含 `research/`，证据日志中必须有对 `research/` 目录的写入操作，否则拒绝
- 文档用 Markdown 格式、中文撰写，专有术语可保留英文

## 方法论

方法论存放在 `methodology/` 目录，顶层索引 `_index.md`。

- **查阅**：任务开始读 `_index.md`，按场景只读匹配的详情文件
- **写入**：反思抽象为方法论，写入对应分类；写入前先查索引避免重复
- **整理**：某分类超 5 个文件时加二级索引；定期合并重叠条目

## 快车道模式（轻量任务豁免）

三次握手是为**会改动项目状态、有风险、多步骤**的任务设计的。对于简单任务（跑一个 skill、单条命令、只读探索、单文件小改），全程走握手 → 等确认 → 写验算太重，也浪费 token。这类任务可以走**快车道**。

### 工具层天然豁免（不用操心）

以下工具**本来就不会被握手拦**：

- **Read / Grep / Glob**：只读探索，随便用。
- **Bash（只读）**：`ls / cat / grep / find / git status / git log / git diff / git commit / git push` 等不改 working tree 的命令，不触发握手检查。
- **Skill 调用**：Skill 工具本身不触发握手；但 Skill 内部若发起 Edit/Write 或 Bash 写操作，仍受握手规则约束。
- **WebFetch / WebSearch / MCP 工具**：不触发握手。

还有**项目目录外的路径**（如 `~/Downloads/foo.md`）—— Edit/Write 直接豁免，不在 discipline 作用域。

### ⚠️ Bash 写操作同样受握手保护（v2.1.0+）

`check-bash-mutation.js` 堵住了"用 Bash 绕开 Edit/Write 握手"的漏洞。以下 Bash 命令**与 Edit/Write 对等**——必须握手完成才能执行：

- **文件操作**：`mv / cp / rm / rmdir / tee / dd / truncate / shred / install`
- **原地编辑**：`sed -i` / `awk -i inplace` / `perl -i`
- **重定向写入**：`> file` / `>> file`（豁免 `/dev/null` 和 `2>&1` fd 复制）
- **Git 破坏性子命令**：`git reset --hard / git clean -fd / git checkout -- / git restore / git rm / git mv`
- **复合命令**：按 `; && || |` 切分后逐段检查——任一段命中 mutation 即触发

不拦的边界命令（低风险，先不拦）：`touch / mkdir / chmod / chown / ln`、`git commit / push / fetch / pull`。

有授权段（含快车道）时这些命令全部放行——与 Edit/Write 行为完全一致。

### 关键词快车道（轻量任务建段简化）

当任务需要 Edit/Write 项目文件但**规模很小**时，可走快车道——把三次握手**压缩成一步**：标题 + 一行执行授权。

**触发条件（满足任一即可）**：

- 用户措辞含：`快速 / 直接 / 小改 / 不用握手 / 一句话 / 跑一下 / 简单 / 轻量`。
- 任务特征：单条命令、单个 skill 调用、只读探索后的小记录、单文件 ≤10 行改动、改错别字、改单个变量名。

**极简任务段模板**：

```markdown
## YYYY-MM-DD — 任务简述 <!-- session: xxxxxxxx -->

**用户意图**：{一句话复述}

> ✅ 执行授权：[快车道] {一句话说明做什么、为什么简单}
```

跳过 AI 理解、验算方案、达标标准、子任务列表。完成后在对话中一句话汇报即可，不强制写 `> ✅ 验算通过` 段。

### 必须走重档（不适用快车道）

以下场景**即使用户说"快速"也必须走完整握手**：

- ❌ 多文件改动（>1 个文件，或单文件 >10 行）。
- ❌ 研究 / 调研 / 分析（产出必须写入 `research/`）。
- ❌ 重构 / 架构调整 / 目录结构变更。
- ❌ 配置变更 / 基础设施 / CI / 依赖升级。
- ❌ 删除文件、数据库 / 外部服务状态变更、任何难以回滚的操作。
- ❌ 用户质疑你对任务的理解 → 立即回退重档，走完整 AI 理解。

**判断法则**：**如果验算需要多步或需要用户参与判断，就不是快车道。** 快车道任务应该"一眼看完结果就知道对错"。

## 严禁的反模式

- ❌ 跳过三次握手直接开干
- ❌ 回复"收到"就开始做，没有回声式复述理解
- ❌ 做完再补 todo（事后补账）
- ❌ 批量标记 `[x]`（攒到最后一起标）
- ❌ 自己说 done 就结束，不等用户验收
- ❌ 验算用和执行相同的路径（读一遍代码≠验算；跑测试才是）
- ❌ 做完才想验算方案（必须在握手时就规划好）
- ❌ 把应走重档的任务（多文件改、研究、重构、配置变更）硬塞进快车道
- ❌ 用 Bash 的 `mv / sed -i / cp / rm / 重定向 / git reset --hard` 绕开握手改项目文件（v2.1.0+ 已由 check-bash-mutation 堵死，但此反模式本身就是纪律违规）
